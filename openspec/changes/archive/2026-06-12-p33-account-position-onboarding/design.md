## Context

P33 是 P32 之后的第一个功能阶段。P32 已提供每日纪律报告入口，但空库用户仍缺少通过 UI 建立本地账户事实的完整路径。当前已有 `portfolio_snapshots`、`positions`、`position_snapshots`、`position_transactions`、`operation_confirmations` 和审计表，可支撑追加式账户事实；P33 应优先复用这些表与事务模式，只有现有字段无法表达导入批次、修正原因或校验结果时才增加轻量结构。

本阶段必须继续保持本地、只读外部源、人工复核和非交易边界。所有账户变化来自用户录入的线下事实，不连接券商账户，不代表系统执行交易。

## Goals / Non-Goals

**Goals:**

- 让空库用户通过本地 UI 完成账户初始化与持仓录入。
- 支持持仓新增、编辑、删除前置确认和校验，历史以追加式快照、交易流水和审计表达。
- 支持用户录入已在线下完成的交易流水，并校验现金、持仓和账户快照一致性。
- 支持批量导入或表格化录入，返回逐行校验结果。
- 支持录入错误修正，记录修正原因和前后引用。
- 让 Dashboard / 今日纪律报告展示首次使用引导和缺前提类别。

**Non-Goals:**

- 不接券商交易 API、券商账户同步、登录源、付费源、授权源、Level2 或高频行情。
- 不新增自动交易、一键买卖、撤单、改单或外部推送。
- 不承诺收益，不预测确定涨跌。
- 不让 LLM 或前端表单覆盖规则裁决。
- 不重构 P0–P32 已稳定的核心模型，除非 P33 必需。

## Decisions

1. **以快照和流水双轨表达本地账户事实。**
   - 初始化和校准写完整 `portfolio_snapshots`、`positions`、`position_snapshots`。
   - 已线下成交写 `position_transactions`，再生成新的账户/持仓快照。
   - 原因：快照便于每日纪律读取，流水便于追溯变化来源。

2. **所有修改都走事务服务。**
   - 单次初始化、单次交易流水、批量导入批次均在事务内写账户、持仓、快照、审计。
   - 任一校验失败时不写入部分事实。
   - 原因：P33 的核心价值是避免账户、现金和持仓不一致。

3. **删除持仓采用“置零/归档式事实”，不做物理删除历史。**
   - 用户删除当前持仓时，写入确认、交易或修正事实，并让最新快照中该持仓数量为 0 或不再作为当前持仓展示。
   - 历史 `position_snapshots` 和审计保留。

4. **批量导入先校验、后确认写入。**
   - 前端可表格化录入或粘贴 CSV；后端返回逐行错误和汇总。
   - 只有用户确认后的有效批次才写入事实。

5. **首次使用引导只展示缺前提，不生成正式建议。**
   - 空库、无持仓、缺行情、缺证据时，Dashboard / 今日报告展示缺前提类别和下一步入口。
   - 不创建伪造的 `decision_records`。

## Risks / Trade-offs

- [Risk] 表单自由度过高导致无效账户事实。→ 使用后端统一校验：金额非负、数量非负、总资产与现金/持仓市值容差、必填买入原因、日期格式。
- [Risk] 编辑持仓被误解为改写历史。→ 前端文案和审计说明显示“生成新快照”，历史保留。
- [Risk] 批量导入造成部分写入。→ 校验与写入分两步，确认写入使用单事务。
- [Risk] 已线下交易被误解为系统交易。→ API、UI 和审计统一使用“记录线下动作”语义，不出现自动交易入口。
- [Risk] P33 范围扩大到完整税费/分红/拆分。→ 首轮只覆盖买入、卖出、减少、现金校准和错误修正；复杂公司行为后续单独规划。
