# P40 Design

## Overview

P40 聚焦本地交付和恢复演练，不新增投资决策能力。实现应优先复用现有 `cmd/agent`、配置加载、SQLite 迁移、备份恢复、health/status API、P39 E2E fixture 和前端运维状态组件，把分散的检查和恢复动作串成可重复流程。

## Proposed Approach

1. **本地预检 / 初始化入口**
   - 提供命令或脚本检查 Go、Node、npm、Playwright browser、SQLite、VecLite、配置文件、数据目录和写权限。
   - 输出分级诊断：pass / warning / failed / skipped。
   - 对缺失项给出本地修复提示，不输出密钥原文或私有路径之外的敏感内容。

2. **启动前诊断**
   - 在本地 server/agent 启动前复用配置校验和迁移检查。
   - 对 SQLite 不可写、VecLite 缺失、数据源配置缺失、LLM 未配置等情况返回稳定错误码或状态。
   - 诊断只写本地日志/审计/诊断文件，不触发交易、推送或规则应用。

3. **备份恢复演练**
   - 基于现有备份能力增加恢复 smoke：从临时备份恢复到临时 DB，运行迁移/health/API 读取。
   - 恢复默认写入临时路径；若目标已有 DB，必须显式确认或使用安全选项，避免覆盖真实本地库。
   - smoke 验证恢复后的决策、审计、持仓或报告事实可读。

4. **数据源健康面板**
   - 复用 P34 source health 和现有设置/复盘页面 DTO。
   - 展示最近成功时间、最近失败时间、失败分类、新鲜度和影响标的。
   - 状态面板只展示和导航，不发起外部推送或自动恢复动作。

5. **产物与 gitignore 治理**
   - 统一记录 `tmp/`、Playwright output、诊断文件、备份演练输出和本地日志位置。
   - 确认生成物被 `.gitignore` 覆盖或在脚本结束时清理。

## Risks

- 预检可能误判不同用户环境。缓解：检查项应分级，并允许跳过非必需依赖。
- 备份恢复可能误覆盖本地数据。缓解：默认临时路径，真实目标恢复必须显式确认。
- 健康面板可能被误解为自动恢复。缓解：所有动作保持查看/本地记录语义。

## Verification

- 后端单元测试覆盖预检、诊断和恢复安全边界。
- 前端测试覆盖健康面板成功、失败、降级和空态。
- smoke 使用临时路径验证恢复后 API/浏览器可读。
