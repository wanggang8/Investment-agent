# P50 设计：后续治理与验收路线图

## 设计目标

P50 解决的是顺序和验收标准问题，不解决运行时功能问题。设计原则：

1. **先补证据，再谈发布**：P19-P24 的历史归档缺口必须先转化为可核验证据包，避免发布材料引用不可追溯结论。
2. **验收门禁分层**：把单元、集成、E2E、真实源、真实 LLM、冒烟、安装诊断、发布升级检查区分为不同门禁，明确各自是否必须、是否可降级、是否需要显式配置。
3. **不伪造历史**：P51 只能整理当前仓库事实、现有文档、现有测试和可重跑命令，不补写历史完成时间，不把缺 archive 改写为已 archive。
4. **发布后置**：P53 只有在 P51/P52 完成后才进入，避免发布材料先于验收体系。

## 后续阶段候选

### P51 `p51-p19-p24-audit-evidence-pack`

目标：为 P19-P24 建立审计证据包。

预期输出：

- `docs/p19-p24-audit-evidence-pack.md` 或同等审计文档。
- 每阶段一张矩阵：交付边界、当前事实源、代码/测试/文档证据、可重跑命令、缺口、不可声明事项。
- 明确 P14-P18 已有 archive，P19-P24 缺少逐阶段完整 archive；若用户请求“P19-P14”，默认解释为 P14-P19 范围核查，其中 P14-P18 仅复核 archive 存在性，重点仍是 P19 之后缺口。
- 不创建伪历史 `p19-*` 到 `p24-*` archive 包。

### P52 `p52-project-acceptance-gate-matrix`

目标：建立项目验收矩阵和发布前门禁。

预期输出：

- 验收矩阵覆盖：
  - 单元测试：核心规则、服务、解析器、数据模型、脱敏。
  - 集成测试：SQLite、API handler、工作流、collector fixture。
  - E2E 测试：前端关键页面、空库到日报、降级路径、窄屏/a11y/console。
  - 真实测试：公开源、真实 LLM、真实本地配置；必须显式 opt-in，失败需分类为网络、限流、解析、凭证、模型不可用等。
  - 冒烟测试：本地启动、诊断、备份恢复、数据源质量、发布升级检查。
  - 安全/边界测试：无券商接口、无自动交易、无外推、无自动确认、无自动规则应用、无 key/raw JSON 泄露。
- 每项门禁定义命令、前置条件、通过标准、允许降级条件、产物位置、是否阻断发布。
- 明确“真实测试通过”不能等价于收益承诺或未来可用性承诺。

### P53 `p53-release-candidate-materials`

目标：在 P51/P52 完成后整理发布候选材料。

预期输出：

- RC 检查清单。
- 本地安装/升级/恢复说明摘要。
- 已知限制与不可声明能力列表。
- 验收结果汇总入口。

## 验收策略

P50 本身为治理变更，验收限定为文档一致性：

- OpenSpec 严格校验通过。
- 进度文档指向 P51。
- 文档不把 P19-P24 缺 archive 状态写成已 archive。
- 文档不新增运行时承诺。

P52 将承接真正的项目级验收设计，并把“单元 / 集成 / E2E / 真实 / 冒烟 / 发布前门禁”固化为可执行清单。

## 风险与缓解

- 风险：过早做发布材料导致验收薄弱。缓解：P53 依赖 P51/P52。
- 风险：历史审计被误写成历史归档。缓解：P51 明确只做证据包，不补写 archive。
- 风险：真实测试不稳定阻塞所有开发。缓解：P52 将真实测试设为显式 opt-in，并要求失败分类和可接受降级边界。
