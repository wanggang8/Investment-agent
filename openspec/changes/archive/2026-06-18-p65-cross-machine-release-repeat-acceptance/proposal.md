# Proposal: P65 Cross-Machine Release Repeat Acceptance

## Why

P64 made the release package reproducible on the development machine, but the project still needs evidence that an operator can take the package, move away from the active repository working tree, and repeat the minimum acceptance path from a clean handoff context.

P65 closes that delivery gap by defining and executing a cross-machine-equivalent repeat acceptance flow. The flow should create a fresh P65 candidate archive using the P64 package workflow, validate that package and its sidecar manifest, extract it into an isolated directory, install/build/test from the extracted source, run a local smoke path, and write a sanitized repeat acceptance record.

This phase is still release verification. It must not add runtime product features, broker connectivity, remote publishing, Git tag creation, automatic upgrades, migrations, restores, repairs, external pushes, or trading behavior.

## What Changes

- Add a local repeat acceptance script that starts from a package archive and sidecar manifest generated by the P64 package workflow for the current P65 candidate.
- Run package verification before extracting.
- Extract into a fresh `tmp/` repeat directory and execute commands from that extracted source, not from the active repository checkout.
- Install frontend dependencies in the extracted workspace using lockfile-based commands.
- Run a minimum repeat gate: OpenSpec validation, Go tests, frontend tests, frontend build, and local E2E smoke with temporary ports and temporary data.
- Write a sanitized repeat summary JSON and release document that records commands, results, package identity, caveats, and safety boundaries.
- Update release handoff and governance progress so P65 is trackable.

## In Scope

- `scripts/local-release-repeat-acceptance.sh`
- P65 release repeat acceptance record under `docs/release/acceptance/`
- Updates to `docs/release/README.md`, `docs/release/release-handoff-2026-06-18.md`, `docs/development-plan.md`, `docs/README.md`, `docs/GOVERNANCE.md`, `AGENTS.md`, `openspec/project.md`, and `openspec/PROGRESS.md`
- OpenSpec delta for the local release-upgrade/release-package repeat acceptance contract
- Real local execution against an extracted package workspace
- Sanitized summary artifacts under `tmp/`

## Out Of Scope

- Physical second-machine execution as a mandatory gate; P65 provides an equivalent isolated local repeat and a handoff command set for physical-machine follow-up.
- Remote publishing, package registry upload, cloud storage upload, app store packaging, installer signing, or Git tag creation.
- Automatic upgrade, migration, restore, repair, rollback, or overwrite of a real user database.
- Broker connectivity, automatic trading, one-click trading, order delegation, external push, automatic confirmation, automatic rule application, or return promises.
- New public data sources, login-gated sources, paid sources, authorization-gated sources, Level2 data, high-frequency data, new HTTP APIs, new SQLite schema, new Eino workflow, or new LLM behavior.

## Validation

- Plan review by sub agent before execution.
- `openspec validate p65-cross-machine-release-repeat-acceptance --strict`
- `openspec validate --all --strict`
- `git diff --check`
- Build a fresh P65 candidate package from current `main` using the P64 package workflow.
- Run package verify against that package.
- Run `scripts/local-release-repeat-acceptance.sh` against that package.
- Confirm repeat summary is sanitized and does not contain complete keys, private paths, raw prompts, raw provider payloads, SQLite databases, logs, traces, or local config.
- Run `go test ./...`, `npm --prefix web test`, `npm --prefix web run build`, and `bash scripts/e2e-smoke.sh` in the active repository as regression gates.
- Execution review by sub agent before archive.
- Submit review by sub agent before commit.
