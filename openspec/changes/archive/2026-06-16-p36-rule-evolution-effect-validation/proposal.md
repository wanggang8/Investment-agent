## Why

P22/P23 已让规则提案和复盘具备生成、审批与追踪基础，P35 又补齐了风险预警状态；下一步需要让规则进化从“能提出”升级到“能解释、能验证、能追踪效果”。P36 用本地历史事实评估提案来源、样本代表性、过拟合风险和应用后效果，同时继续保持守门人审计、用户最终确认和不自动交易边界。

## What Changes

- 新增规则进化效果验证能力：基于错误案例、决策记录、确认结果、复盘摘要、风险预警和审计事件生成提案有效性评估。
- 增强规则提案：记录来源解释、修改前后对比、影响范围、风险说明、样本代表性、过拟合检查和历史回放验证结果。
- 增强守门人审计：样本不足、代表性不足、过拟合风险高或历史回放不利时，拒绝或要求用户复核，不允许进入最终应用。
- 增强规则应用后追踪：追踪命中率、误判率、缺证据率、降级情况和相关风险预警变化。
- 新增本地只读 API 与前端展示：在规则提案、复盘或规则治理页面展示有效性趋势、验证结果和应用后追踪，不提供自动应用或交易动作。
- 更新 OpenSpec 与 L1 文档 delta，归档时同步到 `docs/api.md`、`docs/data-model.md`、`docs/workflow.md`、`docs/frontend-contract.md`。

## Capabilities

### New Capabilities
- `rule-evolution-effect-validation`: 规则提案来源解释、样本代表性、过拟合检查、历史回放验证、应用后效果追踪和本地只读展示。

### Modified Capabilities
- `evolution-proposal-hardening`: 规则提案与守门人审计需消费 P36 的验证结果，维持样本门槛、审计和最终确认边界。
- `review-automation-delivery`: 复盘输出需包含规则有效性趋势、提案应用后追踪和安全追踪入口。

## Impact

- 后端：新增或扩展 rule evolution effectiveness service、DTO、repository、SQLite migration、handler、workflow/local task 聚合逻辑。
- 数据模型：新增本地效果验证/追踪事实表或扩展现有提案 JSON 字段；写入必须可审计、可回放、可降级。
- API：新增规则提案有效性详情、应用后追踪或复盘规则效果查询；统一响应信封和错误分类。
- 前端：规则提案详情、复盘页或规则治理页展示验证结果和趋势。
- 安全边界：不接券商 API、不自动应用规则、不自动交易、不外部推送、不承诺收益、不预测确定涨跌；LLM 不覆盖最终规则裁决。
