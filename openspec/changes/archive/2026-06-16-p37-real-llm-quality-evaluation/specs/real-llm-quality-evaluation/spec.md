## ADDED Requirements

### Requirement: Real LLM configuration SHALL be locally testable
The system SHALL support local configuration of LLM base URL, API key, model name, timeout, and real smoke behavior so a developer can validate analyst material against a real model without changing source code.

#### Scenario: Real model is configured from local config
- **WHEN** the local config contains a base URL, API key, and model name
- **THEN** the LLM client SHALL call the configured model and endpoint
- **AND** it SHALL NOT require environment variables for this path.

#### Scenario: Local key is not committed
- **WHEN** the repository is staged for commit
- **THEN** real local config files containing keys SHALL remain untracked or ignored
- **AND** committed example config SHALL contain only placeholders.

### Requirement: LLM calls SHALL expose quality metadata
The system SHALL record prompt version, input summary, output summary, parse status, quality status, model name, and stable error category for LLM calls.

#### Scenario: LLM analysis succeeds
- **WHEN** the configured LLM returns usable analyst material
- **THEN** the system SHALL expose analysis material, prompt version, input summary, output summary, parse status, quality status, and model name
- **AND** the final verdict SHALL still be generated by the domain rule engine.

#### Scenario: LLM call fails
- **WHEN** the key is missing, a timeout occurs, the endpoint returns an error, the response is empty, the response cannot be parsed, or quality checks fail
- **THEN** the system SHALL classify the failure as missing_key, timeout, http_error, empty_response, parse_error, quality_failed, or unavailable
- **AND** the workflow SHALL degrade without blocking rule arbitration where local facts are sufficient.

### Requirement: LLM output quality SHALL be checked
The system SHALL evaluate LLM output before treating it as analyst material.

#### Scenario: Output contains prohibited claims
- **WHEN** LLM output promises returns, predicts deterministic price movement, instructs direct trading, or attempts to write the final verdict
- **THEN** the output SHALL be marked quality_failed
- **AND** it SHALL NOT be used as final verdict, trading instruction, automatic confirmation, or rule update.

#### Scenario: Output is normal analysis material
- **WHEN** LLM output explains risks, assumptions, evidence, and uncertainty without prohibited claims
- **THEN** the output SHALL be accepted as analyst material only
- **AND** the rule engine SHALL remain responsible for final verdict.

### Requirement: LLM audit SHALL be sanitized
The system SHALL audit LLM invocation summaries without exposing secrets or unnecessary sensitive local facts.

#### Scenario: LLM audit event is written
- **WHEN** an LLM call succeeds, degrades, or fails
- **THEN** the audit summary SHALL include prompt version, model name, input summary, output summary, parse status, quality status, and error category where applicable
- **AND** it SHALL NOT include the API key, full prompt containing unnecessary sensitive data, broker/account secrets, or local filesystem paths.

### Requirement: Real LLM smoke SHALL prove the configured path
The system SHALL provide a real LLM smoke command or test entry that uses local config and verifies the configured endpoint can produce analyst material.

#### Scenario: Real smoke succeeds
- **WHEN** the smoke is run with the local test config
- **THEN** it SHALL call the configured real endpoint and model
- **AND** it SHALL verify analyst material, prompt version, parse status, quality status, and non-trading safety boundaries.

#### Scenario: Real smoke has no trading side effects
- **WHEN** real smoke runs
- **THEN** it SHALL NOT create or mutate positions, portfolio snapshots, operation confirmations, position transactions, rule versions, broker state, orders, or external notifications.
