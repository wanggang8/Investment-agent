## Context

P0-P9 已完成本地验收，当前系统具备 SQLite、规则引擎、HTTP API、前端页面、本地任务和审计主链路。对照总需求、架构、工作流和前端契约后，产品级缺口集中在真实数据、真实检索、完整工作流节点、前端操作入口和文档状态一致性。

约束：

- 禁止自动交易。
- DeepSeek 只生成分析材料，不写最终裁决。
- SQLite 仍是事实基准，VecLite 只能作为可重建辅助索引。
- P10 不新增 development-plan 以外的产品方向，只补齐已定义但未完全产品化的能力。

## Goals / Non-Goals

**Goals:**

- 将真实行情和情报源从 stub 可替换推进到可配置真实适配与失败降级。
- 将 VecLite 从内存/适配实现推进到 JSON 文件索引适配层，保证文件持久化、重建、检索与降级可验证。
- 将 EvidenceVerificationGraph 和 Daily/Consultation 工作流推进到节点级 Eino 编排。
- 补强守门人审计的规则检查、冲突检查、回测样本和审计理由。
- 让前端具备刷新市场、重建索引、更新设置、账户录入、规则库展示、证据验证面板和单条决策审计时间线。
- 对齐 `docs/development-plan.md` 与已归档 P9 状态。

**Non-Goals:**

- 不接入自动下单或券商交易 API。
- 不让系统主动推荐具体标的。
- 不提供收益承诺或确定性涨跌预测。
- 不把 VecLite 作为不可重建事实来源。

## Decisions

1. 真实数据源采用接口适配加配置开关。
   - 理由：保留本地 stub 验收能力，同时允许接入真实行情、公告、新闻和手工导入。
   - 替代方案：删除 stub。未采用，因为会让本地开发依赖外部服务。

2. VecLite 以文件索引适配器接入，SQLite 摘要仍为事实基准。
   - 理由：契约要求 VecLite 可重建，索引损坏时可从 `rag_chunks` 与 `intelligence_summary` 恢复。
   - 当前 P10 实现先提供 JSON 文件 chunk 索引适配层，用于验证配置路径、持久化、重建、检索、损坏降级和审计；真实 VecLite 向量库 API 可在供应商/API 明确后替换该适配层。
   - 替代方案：只保留内存索引。未采用，因为无法证明文件备份、恢复和重建能力。

3. Eino Graph 拆为节点级编排，但保留现有步骤函数作为节点实现。
   - 理由：减少重写风险，同时满足工作流文档中的节点可审计要求。
   - 替代方案：完全重写工作流。未采用，因为会扩大变更范围。

4. 前端操作入口只触发刷新、重建、设置更新和线下记录，不触发交易。
   - 理由：补齐可用性，同时保持安全边界。
   - 替代方案：引入交易执行入口。禁止采用。

5. 文档状态修正放在 P10 任务内完成。
   - 理由：P9 已归档但 `docs/development-plan.md` 仍有未勾选项，需在有 OpenSpec change 的情况下对齐。

## Risks / Trade-offs

- 真实外部数据不稳定 → 保留 stub、本地手工导入和降级错误码。
- VecLite 真实向量库 API 未固定 → P10 先以 JSON 文件适配层验证可重建索引边界，后续可替换为真实 VecLite 实现。
- Eino 节点拆分影响现有测试 → 复用现有步骤函数，并要求旧工作流测试继续通过。
- 前端操作入口可能让用户误解为可交易 → 所有文案明确“只刷新/记录/配置，不交易”。
- 守门人审计数据样本不足 → 样本不足时返回 `needs_user_review` 或阻止应用。

## Migration Plan

1. 对齐开发计划状态，只勾选已由归档和测试证明完成的 P9 条目。
2. 补真实数据源和 VecLite 文件索引，保留 stub 开关。
3. 拆分 Eino 节点并保留原有工作流输出。
4. 补前端操作入口和详情展示。
5. 执行 Go、前端、OpenSpec 和本地任务验收。
