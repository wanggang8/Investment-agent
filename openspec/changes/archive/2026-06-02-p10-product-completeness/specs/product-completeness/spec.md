## ADDED Requirements

### Requirement: P10 development plan status alignment
The system documentation MUST align `docs/development-plan.md` P9 task status and total acceptance checklist with archived P9 completion evidence before new product-completeness work is implemented.

#### Scenario: P9 archived tasks reflected in development plan
- **WHEN** P10 documentation alignment is performed
- **THEN** P9.1, P9.2, P9.3 items already completed by `p9-review-automation-delivery` MUST be marked complete in `docs/development-plan.md`

#### Scenario: Total checklist reflects completed P9 boundaries
- **WHEN** the development acceptance checklist is reviewed
- **THEN** completed entries for `cmd/agent`, passive symbol boundary, and monthly/quarterly review boundary MUST match the archived implementation state

### Requirement: Product-grade real data and retrieval
The system SHALL support configurable real market data, intelligence ingestion, a rebuildable JSON file indexing adapter for VecLite replacement, index rebuild, and fallback to SQLite summaries without treating any local index as the source of truth.

#### Scenario: Real source unavailable
- **WHEN** a configured real market or intelligence source is unavailable
- **THEN** the system MUST return the existing stable error or degraded state and MUST record an audit event

#### Scenario: Local file index unavailable
- **WHEN** the JSON file index adapter is missing, corrupted, or incompatible
- **THEN** the system MUST rebuild from SQLite `rag_chunks` and `intelligence_summary` when possible, or return information-insufficient state when SQLite summaries are insufficient

### Requirement: Product-grade workflow orchestration
Daily discipline, consultation, evidence verification, and gatekeeper audit workflows SHALL use node-level Eino Graph orchestration while preserving rule-first final arbitration and audit traceability.

#### Scenario: Consultation workflow executes as node-level graph
- **WHEN** a consultation request is processed
- **THEN** state snapshot, capability check, evidence retrieval, analyst material, expected return, rule arbitration, and decision record MUST be represented as auditable graph nodes

#### Scenario: Analyst output cannot decide final verdict
- **WHEN** DeepSeek returns analysis material
- **THEN** the final verdict MUST still be generated by the domain rule engine

### Requirement: Product-grade frontend operation surfaces
The frontend SHALL expose safe local operation surfaces for data refresh, index rebuild, settings update, account entry, rule library viewing, source verification details, and decision audit timeline without adding any automatic trading entry.

#### Scenario: User refreshes data from frontend
- **WHEN** the user triggers market refresh or evidence index rebuild
- **THEN** the frontend MUST call the corresponding API, show request status, and state that no trade is executed

#### Scenario: User reviews a decision detail
- **WHEN** the user opens a decision detail page
- **THEN** the page MUST show the decision trace and related audit timeline or a linkable embedded audit section

### Requirement: Product-grade validation
P10 completion MUST be verified with Go tests, frontend tests/build, OpenSpec validation, and local task checks.

#### Scenario: P10 validation is complete
- **WHEN** P10 tasks are marked complete
- **THEN** `go test ./...`, frontend test/build commands, OpenSpec strict validation, and `cmd/agent` local task checks MUST have recorded results
