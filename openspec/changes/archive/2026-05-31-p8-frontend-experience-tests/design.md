# Design: P8 前端体验与测试

## Context

P7 已完成真实数据、RAG/VecLite 与 DeepSeek 分析材料的后端底座。当前前端已有驾驶舱页面、状态展示、用户确认、规则与审计等基础能力。P8 在不改变后端契约和产品边界的前提下，增强图表、筛选、展开交互、空态/错误态和前端测试。

约束：

- 前端数据只来自 API DTO 与前端契约，不直接访问 SQLite、VecLite 或本地文件。
- 用户确认区只记录线下动作，不能增加自动交易或一键交易入口。
- 规则提案仍需守门人审计和用户最终确认，前端不得自动应用。
- 非显然实现逻辑写中文注释，特别是图表数据归一化、状态映射、禁止交易入口断言等位置。

## Goals / Non-Goals

**Goals:**

- 为今日纪律、持仓、复盘页面增加基于 API DTO 的图表展示。
- 增强证据、决策链、审计时间线的筛选和展开交互。
- 对信息不足、数据过期、LLM 降级、VecLite 不可用、错误响应等状态提供明确 UI。
- 建立前端测试脚本，覆盖 API client、关键状态渲染、用户确认流程、规则提案最终确认流程和无自动交易入口断言。
- 确保 `npm run build` 与 `npm test` 成为 P8 验收命令。

**Non-Goals:**

- 不新增后端 API、不修改 L1 API 契约。
- 不实现 P9 本地任务入口、复盘自动化和交付说明。
- 不接入真实图表外部服务或远程分析平台。
- 不新增自动交易、一键交易、代下单或收益承诺能力。

## Decisions

### 1. 图表使用轻量前端组件，不引入复杂外部依赖

优先使用现有 React + CSS/SVG 实现基础图表：仓位占比、风险状态、证据覆盖、复盘摘要趋势。这样可以减少依赖变化，也能让测试更容易断言。若现有项目已经有图表库，再按既有库扩展。

备选方案是引入大型图表库。该方案展示能力更强，但会增加构建体积和测试复杂度，P8 当前没有此需求。

### 2. 图表数据在前端通过 DTO 适配层转换

新增或扩展前端适配函数，把 API DTO 转成图表组件需要的展示模型。图表组件只接收展示模型，不关心接口细节。适配函数中对非显然映射写中文注释。

该方式避免组件直接依赖后端字段散落在页面里，也便于测试信息不足、降级和错误状态。

### 3. 状态 UI 使用统一状态映射

对 `insufficient_data`、`frozen_watch`、`high_risk`、`DATA_STALE`、`VECTOR_INDEX_UNAVAILABLE`、`ANALYST_UNAVAILABLE` 等状态建立统一映射，页面复用状态卡片或空态组件。

这样能减少不同页面文案不一致，也能让测试集中覆盖关键状态。

### 4. 前端测试优先覆盖契约行为

测试重点放在：

- API client 正确处理 `request_id`、`data`、`error`。
- 页面正确展示关键状态、错误响应和降级信息。
- 用户确认流程只记录线下动作。
- 规则提案 `pending_final_confirm` 可见且不会自动应用规则。
- 页面没有自动交易、一键交易或代下单入口。

测试不要求覆盖所有视觉细节，避免把样式变动变成脆弱测试。

## Risks / Trade-offs

- 图表展示与后端 DTO 字段不一致 → 通过前端类型和测试 fixture 固定契约样例。
- 空态和错误态散落各页面 → 使用共享状态组件和映射函数。
- 测试过度绑定文案 → 测试核心语义和关键按钮，不依赖长文案全文。
- 无自动交易断言遗漏 → 在关键页面和确认流程测试中统一检查禁止入口。
- 图表组件过度复杂 → P8 只实现计划要求的仓位、风险、证据覆盖、复盘摘要展示。

## Migration Plan

1. 确认可复用的前端页面、API 类型和状态组件。
2. 增加图表展示模型与组件。
3. 在今日纪律、持仓、复盘页面接入图表。
4. 增强证据、决策链、审计时间线交互。
5. 增加空态、错误态和降级态展示。
6. 增加前端测试配置与测试用例。
7. 执行 `cd web && npm run build && npm test`。
