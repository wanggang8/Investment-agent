## Context

P0–P32 已完成或归档，P33–P40 是当前 `docs/development-plan.md` 中剩余的计划内产品化阶段。P33 依赖 P32 形成的每日纪律报告入口，P39 依赖 P33 提供空库到首份报告的真实用户旅程，P34/P35/P38 共同提升数据、风险和证据质量，P36/P37/P40 收尾规则进化、LLM 质量和本地交付体验。

当前治理约束保持不变：实现前必须存在活跃 OpenSpec change；L1 契约修改只能通过 change delta；P19–P24 若需 archive 级追溯，需要独立治理 change，不伪造历史归档。

## Goals / Non-Goals

**Goals:**

- 固化 P33–P40 为当前计划内剩余功能队列。
- 明确推荐执行顺序、阶段依赖和每阶段验收方向。
- 明确 P32 已归档，下一步先完成路线图治理，再启动 P33。
- 明确 P40 后如果继续扩展，需要新路线图 change。

**Non-Goals:**

- 不在本 change 实现 P33–P40 任一功能。
- 不修改券商、交易、外部推送、登录源、付费源、授权源、Level2 或高频数据边界。
- 不把 P19–P24 历史已交付状态伪造成完整 archive 包。
- 不把 `docs/superpowers/plans/` 作为规格或验收真源。

## Decisions

1. **按三组推进 P33–P40。**
   - 第一组：P33 → P39。先补齐账户/持仓初始化，再做浏览器级完整用户旅程。
   - 第二组：P34 → P35 → P38。先增强真实公开数据，再建立风险预警体系，并加固证据检索质量。
   - 第三组：P36 → P37 → P40。再完善规则进化效果验证、真实 LLM 质量评估和本地交付演练。
   - 原因：用户可用路径优先，数据和风险质量其次，交付与质量治理作为最终固化层。

2. **P33 是 P32 之后的第一个功能阶段。**
   - P32 让每日纪律报告可见；P33 让空库用户能提供账户和持仓事实。
   - P33 完成前，P39 的“从初始化到首份报告”只能部分验收。

3. **P34/P35/P38 保持只读与可降级边界。**
   - P34 只接公开、低频、可降级数据。
   - P35 只生成风险状态、通知、审计和人工复核提示。
   - P38 只提升检索质量与引用一致性，不绕过 source verification 和规则裁决。

4. **P36/P37 不改变最终裁决归属。**
   - P36 的规则提案效果验证仍需守门人审计和用户最终确认。
   - P37 的 LLM 只生成分析材料，不能写最终 verdict，不能触发交易。

5. **历史追溯与新路线图分离。**
   - P19–P24 archive 追溯如有需要，单独建治理 change。
   - P40 后新增产品愿景，也单独建下一轮路线图 change。

## Risks / Trade-offs

- [Risk] P33–P40 同时展开会造成跨阶段耦合。→ 每个阶段独立 change，按 tasks 验收后再推进。
- [Risk] 真实数据扩展导致数据质量参差。→ P34 必须定义新鲜度、缺失、过期和失败分类，缺数据时不得伪造。
- [Risk] 风险预警文案被误读为交易指令。→ P35 前端必须展示建议人工动作和禁止动作，保留非交易声明。
- [Risk] LLM 质量评估被误当裁决质量。→ P37 只评估分析材料，规则裁决仍由领域规则产生。
- [Risk] P40 后仍有愿景未拆分。→ 当前只承诺 P33–P40；P40 后使用新治理 change 继续规划。
