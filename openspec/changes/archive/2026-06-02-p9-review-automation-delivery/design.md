## Context

P0-P8 已完成并归档，系统具备 SQLite 事实存储、Eino 工作流、HTTP API、前端驾驶舱、真实数据/RAG/DeepSeek 底座和前端测试。P9 需要把这些能力组织成本地可操作的长期使用流程：用户可以通过 `cmd/agent` 手动触发任务，通过前端查看周期复盘和规则建议，并依据文档完成本地部署、备份、重建和故障处理。

约束：

- 禁止自动交易；所有账户变化仍只能来自用户记录的线下动作。
- DeepSeek 仍只能生成分析材料，最终裁决继续由规则引擎负责。
- 规则有效性评估只能产生建议或提案，不能绕过守门人审计和用户最终确认。
- 实现代码中的调度、安全边界和审计写入逻辑需要写中文注释。

## Goals / Non-Goals

**Goals:**

- 提供 `cmd/agent` 本地任务入口，覆盖每日纪律、市场刷新、情报索引和复盘任务。
- 让本地任务执行可审计、失败可读、数据一致性可保持。
- 生成月度/季度复盘摘要，评估规则命中、误判、缺证据和降级情况。
- 将规则有效性评估结果接入规则提案或复盘摘要，并保持既有确认链路。
- 让前端复盘页展示周期摘要、规则建议和追踪入口。
- 补齐本地交付与运维说明，覆盖启动、初始化、备份、重建、恢复和常见故障处理。

**Non-Goals:**

- 不新增券商交易接口、自动下单、一键交易或代下单能力。
- 不改变 P0-P8 已确立的核心数据模型，除非 P9 任务必须使用现有字段补齐查询。
- 不让复盘自动修改规则版本；规则变更仍走提案、守门人审计、最终确认。
- 不发明 development-plan 之外的产品需求。

## Decisions

1. **`cmd/agent` 复用现有应用依赖组装**
   - 入口应复用 server 已有配置、SQLite migration、repository、workflow dependencies。
   - 原因：避免为本地任务建立第二套运行时，减少数据和审计差异。
   - 中文注释重点：命令调度、安全边界、审计写入和失败处理。

2. **本地任务先支持手动触发，调度只做配置与显式入口**
   - P9.1 要求支持手动触发和本地调度配置，但默认不启用自动交易能力。
   - 实现上可以提供配置结构、命令参数或 dry-run 信息，避免后台隐式执行不可见任务。
   - 原因：符合本地优先和审计优先，同时降低误触发风险。

3. **周期复盘以现有事实表聚合，不另建第二套事实来源**
   - 月度复盘从确认动作、错误案例、规则提案和审计事件聚合。
   - 季度复盘从规则命中、误判、缺证据和降级情况聚合。
   - 原因：SQLite 已是唯一事实来源，复盘应可追溯到现有记录。

4. **规则有效性评估只产出建议或提案**
   - 评估结果可写入规则提案或复盘摘要。
   - 规则应用仍需守门人审计和用户最终确认。
   - 原因：延续 P0-P8 的安全边界，避免自动应用规则。

5. **前端复盘页只展示 DTO，不直接访问本地数据文件**
   - 周期摘要、规则建议和追踪入口通过 API/服务层获取。
   - 原因：维持前端契约边界，并便于测试。

## Risks / Trade-offs

- [Risk] 本地任务入口可能被误解为自动执行能力。→ 命令文案、中文注释和测试都明确“只触发本地分析/复盘，不下单”。
- [Risk] 周期复盘聚合口径可能过宽。→ 仅覆盖 development-plan 指定的确认动作、错误案例、规则提案、审计、规则命中、误判、缺证据和降级。
- [Risk] 调度配置可能引入后台不可见行为。→ 默认不自动运行；任务执行必须写审计事件。
- [Risk] 文档可能泄露本地路径、密钥或账户信息。→ 文档只写示例路径和占位符，不写真实密钥、账号、token 或个人敏感信息。
