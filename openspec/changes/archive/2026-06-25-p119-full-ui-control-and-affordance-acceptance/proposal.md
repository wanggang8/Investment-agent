# P119 Full UI Control and Affordance Acceptance

## Why

P114-P118 已覆盖 UI 视觉重构、真实使用场景、复杂边界与产品可用性验收，但用户继续指出按钮错位、表单错位、卡片同层级高低不同、以及“是否所有操作按钮和 UI 都需要验收”的风险。当前需要一个专门的、可重复执行的全页面控件与视觉 affordance 验收层，证明不是只抽查主页或关键流程。

## What Changes

- 建立 P119 全页面 UI 控件盘点与验收矩阵。
- 对当前生产路由逐页采集按钮、链接、输入框、下拉框、文本域、详情展开控件和导航入口。
- 对控件按只读导航、轻交互、真实写操作、危险/确认操作、禁用/不可用、产品边界说明进行分类。
- 对关键可安全自动化的写操作执行真实 UI 点击，并用 API/SQLite 读回验证后端已实现，而不是假按钮。
- 对桌面与移动视口执行布局扫描，检查控件溢出、明显错位、空白页、前端错误、API 5xx、raw/debug/mock/stub/secret 泄露。
- 保留安全负证据：不得新增券商接口、自动交易、一键交易、代下单、外部推送、自动确认、自动规则应用、收益承诺。

## Out of Scope

- 不做发布/安装/升级场景。
- 不新增券商接入、自动交易、一键交易、代下单、外部推送、自动确认、自动规则应用或收益承诺。
- 不声称物理第二机器复验、真实外部 provider 全覆盖、真实 LLM 输出或未来收益准确性。
- 不归档 P114-P118；P119 完成后等待用户确认再归档。

## Acceptance

- P119 runner 必须能在隔离临时 SQLite、Go backend、Vite frontend 下独立启动并完成。
- 每个生产路由必须有桌面控件盘点证据，关键页面必须有移动布局证据。
- 可见控件必须被分类；未分类或无名称的可操作控件为失败。
- 关键写操作必须有 API/SQLite 读回：持仓/线下交易、决策确认、风险 SOP、通知已读、数据质量处置、规则提案、本地知识导入、证据维护或设置刷新。
- 页面不得出现空白、明显溢出、控制台错误、页面错误、API 5xx、raw/debug/mock/stub/secret 泄露。
- `openspec validate p119-full-ui-control-and-affordance-acceptance --strict` 必须通过。
