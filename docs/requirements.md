# 个人 AI 投资纪律 Agent 需求文档

> 文档版本：v3.0  
> 最后更新：2026-06-17
> 设计目标：构建一个遵循投资大师智慧、严格执行个人纪律、具备多视角辩论能力、可量化评估收益预期、能管理持仓状态、并能通过受控反馈持续进化的 AI 投资决策辅助系统。  
> 核心定位：不做预测未来的先知，只做执行纪律的哨兵，同时作为用户的投资数字分身，安全地从经验中学习。

## 1. 背景与目标

### 1.1 产品背景

个人投资者在实际决策中容易受到市场波动、热点情绪、单一消息源和短期盈亏影响，导致追涨、恐慌卖出、仓位失衡、忽视卖出纪律等问题。本系统旨在将用户预设的投资纪律、数据阈值、持仓状态和反馈机制固化为可审计的 AI 辅助流程。

### 1.2 产品目标

- 帮助用户基于预设纪律进行投资决策辅助。
- 将投资大师的抽象智慧转换为可执行规则。
- 对用户主动询问的标的进行多角度分析。
- 提供可解释、可追溯、可量化的建议。
- 通过错误案例、用户确认和安全审计实现受控进化。

### 1.3 非目标

- 不预测未来涨跌。
- 不主动推荐具体标的。
- 不承诺收益。
- 不代替用户做最终买卖决定。
- 不使用未经验证的市场传言作为决策依据。

## 2. 核心原则与行为边界

### 2.1 核心原则

| 原则 | 说明 |
| --- | --- |
| 不预测，只应对 | 只基于当前状态、数据和预设纪律给出应对建议。 |
| 纪律优先 | 规则执行优先于模型自由分析。 |
| 可解释 | 每条建议必须能追溯到数据、规则和推理路径。 |
| 白箱化 | 规则、阈值、SOP、数据来源对用户透明。 |
| 概率表达 | 输出置信区间、情景概率和风险提示，避免绝对判断。 |
| 受控进化 | 任何规则变更必须经过用户确认和安全审计。 |

### 2.2 角色定位

- 纪律执行官：执行用户预设规则。
- 投资思想翻译官：将大师智慧转换为操作纪律。
- 信息侦察兵：从可靠信源获取并筛选信息。
- 决策参谋：提供多角度分析和灰度建议。
- 安全学徒：从错误中归纳经验，但不能自行修改核心规则。

### 2.3 禁止行为

| 禁止行为 | 约束说明 |
| --- | --- |
| 预测涨跌 | 不输出明日涨跌、短期确定性方向等判断。 |
| 推荐具体标的 | 只分析用户主动提出且在能力圈内的标的。 |
| 承诺收益 | 所有收益区间仅为概率参考。 |
| 使用 C 级信源作为正式裁决依据 | 社交媒体、股吧、群聊转发等只能作为 `background` 背景材料，不得作为交易类正式裁决依据。 |
| 极端情绪下建议交易 | 情绪极端时只输出冷静提醒、数据展示和预设计划。 |
| 忽视流动性 | 建议操作时必须检查成交额和流动性风险。 |
| 自动修改铁律 | 根本规则、阈值、SOP 修改必须经过审批。 |

### 2.4 可执行判定标准

以下判定标准用于规则引擎、Eino 工作流和 UI 状态展示。所有标准必须可审计，不能只依赖大模型主观判断。

| 判定项 | 触发条件 | 系统处理 |
| --- | --- | --- |
| 情绪极端 | 情绪指标处于历史 90% 分位以上或 10% 分位以下；或融资余额、成交量、媒体热度连续 3 个交易日异常放大；或用户文本中出现明确恐慌、狂喜、清仓、满仓等表达 | 进入冷静机制，暂停主动交易建议 |
| 流动性风险 | 标的近 20 个交易日日均成交额低于用户计划成交额的 20 倍；或单日计划成交额超过当日成交额 5%；或买卖价差异常扩大 | 禁止市价式大额操作，只允许提示分批、限价或暂停 |
| 买入逻辑破坏 | 原买入理由对应的核心指标失效；或至少 2 个 A 或 S 级独立信源确认基本面、政策、信用、流动性发生重大不利变化；或用户无法复述当前持仓理由 | 持仓进入只卖不买，禁止新增买入和加仓 |
| 能力圈外 | 标的不属于用户配置的资产类型、行业、地区或策略范围；或用户无法说明收益来源、主要风险和退出条件 | 拒绝交易类分析，只允许输出研究前置问题 |
| 数据不足 | 账户未录入、行情过期、估值缺失、S/A/B 级证据不足、VecLite 索引不可用且 SQLite 摘要不足 | 前端进入信息不足状态，不生成交易类建议 |
| 多源验证不足 | 涉及买入逻辑破坏、重大利好或重大利空的信息少于 2 个 A 或 S 级独立信源确认 | 进入冻结观察，等待更多证据 |

### 2.5 情绪指标来源

情绪极端判定使用三类输入，不允许只由大模型主观判断：

| 来源 | 数据 | 用途 |
| --- | --- | --- |
| 市场交易数据 | 成交量、换手率、波动率、涨跌幅、融资余额变化 | 识别市场过热、恐慌和异常波动 |
| 媒体热度统计 | 新闻数量、标题情绪标签、信源等级、发布时间 | 识别媒体高频报道和情绪共振 |
| 用户文本标签 | 恐慌、狂喜、清仓、满仓、追热点等表达 | 识别用户当下行为风险 |

上述数据进入 Eino 工作流的 `market_snapshot`、`WorkflowContext.media_heat_summary` 和 `user_emotion_tags`，由规则裁决节点统一判断是否触发冷静机制。

| 持仓状态 | `normal` / `sell_only` / `frozen_watch` | 单个持仓的纪律状态 |
| 页面状态 | `first_use` / `normal` / `insufficient_data` / `frozen_watch` / `high_risk` | 驾驶舱或页面整体状态 |
| 用户确认状态 | `not_required` / `pending` / `planned` / `executed_manually` / `watch` / `marked_error` | 系统建议当前被用户处理到哪一步 |

## 3. 用户与使用场景

### 3.1 目标用户

- 有长期投资纪律意识的个人投资者。
- 使用 ETF、指数基金、债基、主题 ETF 等工具进行组合配置的用户。
- 希望减少情绪化交易、提升纪律执行能力的用户。

### 3.2 关键场景

- 每日查看持仓、估值、仓位和风险提醒。
- 持仓大涨后判断是否分批止盈。
- 持仓大跌后判断是否继续持有或重新评估买入逻辑。
- 用户主动询问某标的是否值得研究。
- 宏观事件、突发事件发生后进行信息核查和状态管理。
- 月度、季度复盘纪律执行情况。
- 标注错误决策，并生成规则优化提案。

## 4. 系统架构需求

系统分为四层：用户交互层、多 Agent 辩论与学习层、数据处理层、外部数据源层。

### 4.1 用户交互层

需要提供以下能力：

- 以 Agent 决策驾驶舱作为主交互入口，集中展示账户状态、风险红线、触发规则、证据链、Agent 观点、裁决结果和用户确认区。
- 每日纪律报告：展示仓位、PE 分位、红线触发情况。
- 风险预警：提示估值高位、逻辑破坏、止盈信号、流动性风险。
- 决策咨询：用户主动提问后，返回结构化分析报告。
- 错误标注：用户可标记决策结果和原因。
- 提案确认：用户可查看、确认或拒绝规则优化提案。
- 交互边界：系统不提供自动交易入口，不以聊天气泡承载关键建议，不把大模型作为页面主体。

### 4.2 多 Agent 辩论与学习层

| 角色 | 任务 |
| --- | --- |
| 价值分析师 Agent | 从估值、长期价值、安全边际角度分析。 |
| 趋势与风控官 Agent | 从价格趋势、组合风险、仓位上限角度分析。 |
| 信息核查官 Agent | 执行信源分级、多源验证、事实核查。 |
| 裁决机制 | 按预设规则优先级进行裁决，不依赖大模型自由裁量。 |
| 进化提案引擎 | 基于错误案例生成规则优化提案。 |
| 安全审计官 | 对规则修改进行合规性、过拟合和稳健性审计。 |

### 4.3 数据处理层

- 结构化数据中心：使用 SQLite 存储行情、估值、资金流向、财务等数据。
- 非结构化情报管道：使用 RAG 和 VecLite 处理公告、新闻、研报摘要。
- 信源分级模块：实现 S/A/B/C 信源分类和多源验证。
- 账户状态同步模块：使用 SQLite 保存用户确认后的账户快照、持仓、交易流水和操作确认记录。
- 错误案例库：使用 SQLite 存储错误决策和上下文。

### 4.4 外部数据源

| 数据类型 | 建议来源 |
| --- | --- |
| 行情与估值 | Tushare、baostock、AKShare、中证指数官网 |
| 公告与新闻 | 巨潮资讯、交易所、证监会、东方财富、财联社、四大证券报 |
| 资金与情绪 | 东方财富 Choice、同花顺 iFinD |
| 财务数据 | Tushare、巨潮资讯、上市公司公告 |

公开数据源接入必须先验证、后实现。候选源进入生产 collector 范围前，必须记录权威性、公开访问形态、稳定请求或页面证据、可用字段、更新频率、法律声明或访问限制、限频假设、失败行为和目标入库路径；若只存在网页入口但无法确认稳定参数、分页、响应字段或访问约束，该源只能保持调研或阻塞状态，不能作为必需运行时依赖。

P19/P20 的完成范围是可配置 HTTP bridge、payload parser、fixture/stub fallback、信源等级映射和降级行为；真实公开权威源 collector、历史补采和 live smoke 验证属于 P25+ 之后的独立实现范围。

公开数据集成必须保留本地安全边界：默认排除登录、付费访问、券商交易访问、Level2 或授权行情，以及任何需要绕过访问控制的来源；第三方财经门户或聚合源默认只能作为 B 级或背景证据，除非后续变更记录更强的权威性理由，并且不得单独满足高置信正式证据要求。

## 5. 数据底座需求

### 5.1 结构化数据

| 数据类别 | 核心字段 |
| --- | --- |
| 行情数据 | 日期、开盘价、最高价、最低价、收盘价、成交量 |
| 估值数据 | PE、PB、PE 分位、PB 分位 |
| 资金流向 | 日期、资金净流入、资金净流出 |
| 融资融券 | 日期、两融余额、余额变化率 |
| 成分股财务 | 营收、净利润、增速、披露日期 |

### 5.2 非结构化情报

处理流程：

1. 从 S/A/B 级信源定时获取新闻、公告、研报。
2. 对每篇内容提取结构化摘要，包括实体、事件、影响方向、可信度等级、发布时间。
3. 文本摘要向量化后存入 VecLite。
4. 结构化小结存入 SQLite。
5. 决策时优先查询结构化小结，需要详细分析时回溯原文。

建议数据表：

| 表名 | 用途 | 关键字段 |
| --- | --- | --- |
| `intelligence_items` | 保存原始情报与来源信息 | 来源名称、原文 URL、来源等级、抓取时间、内容 hash、发布时间 |
| `intelligence_summary` | 保存结构化情报摘要 | 情报 ID、实体、事件、影响方向、摘要、时效权重、来源等级 |
| `source_verifications` | 保存多源验证结果 | summary_id、验证状态、独立信源数量、最高信源等级、evidence_role |
| `rag_chunks` | 保存可检索文本块索引 | summary_id、chunk 文本、VecLite 向量 ID、index_status、创建时间 |

### 5.3 信源分级

| 等级 | 定义 | 示例 | 用途 |
| --- | --- | --- | --- |
| S 级 | 法定事实 | 交易所公告、证监会文件、央行公告 | 最终事实基准 |
| A 级 | 权威发布 | 巨潮资讯、四大证券报、上市公司公告 | 交叉验证主要依据 |
| B 级 | 专业分析 | 知名券商研报、头部财经媒体 | 参考分析视角 |
| C 级 | 未经证实传言 | 股吧、群聊、自媒体 | 只能作为 `background` 背景材料，不得作为正式裁决依据 |

### 5.4 防伪规则

| 编号 | 规则 |
| --- | --- |
| F-1 | 每条情报必须标注信源等级，无来源信息丢弃。 |
| F-2 | 触发买入逻辑破坏、重大利好或重大利空的重大信息，必须至少有 2 个 A 或 S 级独立信源确认。 |
| F-3 | 涉及财务数字时，以本地结构化财报数据为准。 |
| F-4 | 情报按时效分段降权：0-24 小时权重 1.0，1-7 天权重 0.8，7-30 天权重 0.5，30 天以上权重 0.2 且仅作背景。 |
| F-5 | 情绪化描述必须转换为客观数据描述后再参与分析。 |

## 6. 决策规则需求

### 6.1 根本规则

| 编号 | 规则 | 目的 |
| --- | --- | --- |
| R-1 | 不凭单一信源决策 | 降低假消息导致的错误操作。 |
| R-2 | 不在恐慌或狂喜时决策 | 降低情绪化交易。 |
| R-3 | 不超过仓位上限 | 控制高风险资产敞口。 |
| R-4 | 不在买入逻辑破坏后加仓 | 避免扩大错误。 |
| R-5 | 不满仓，保留现金冗余 | 保留风险应对空间。 |
| R-6 | 警惕流动性风险 | 避免低流动性标的大额市价交易。 |

### 6.2 裁决优先级

当多 Agent 观点冲突时，系统按以下顺序裁决：

1. 买入逻辑是否被破坏：若是，建议卖出并终止辩论。
2. 信源是否满足多源验证：若不满足，暂停决策并等待验证。
3. 市场情绪是否处于极端区间：若是，锁定主动交易，仅输出冷静提醒。
4. 前述条件均不触发时，综合价值分析师和趋势风控官论据，输出灰度建议。

### 6.3 市场状态量化标准

| 市场状态 | PE 分位 | 情绪/资金标准 | 操作建议 |
| --- | --- | --- | --- |
| 高危区 | >80% | 融资余额急升、成交量剧增、媒体高频报道 | 禁止买入，启动移动止盈评估 |
| 观察区 | 50%-80% | 资金小幅波动、情绪中性 | 仅持有，暂停定投 |
| 舒适区 | 30%-50% | 成交量平稳、无极端情绪 | 按计划定投 |
| 低估区 | <30% | 情绪低迷、成交量萎缩、媒体冷清 | 在买入逻辑完好前提下，分批补仓或加大定投 |

### 6.4 移动止盈规则

| 条件 | 建议 |
| --- | --- |
| 浮盈达到 20% | 建议卖出 30% 仓位，并启动移动止盈。 |
| 浮盈达到 30% | 建议再卖出 30% 仓位。 |
| 剩余仓位 | 设置 10% 回撤步长，从高点回撤 10% 时触发清仓建议。 |
| PE 分位 >80% 且浮盈 >20% | 止盈信号优先级最高。 |

### 6.5 持仓状态机

| 状态 | 说明 | 允许动作 |
| --- | --- | --- |
| 正常持有 | 买入逻辑完好，正常监控。 | 持有、定投、分批调整 |
| 只卖不买 | 买入逻辑确认破坏。 | 卖出，不允许买入或加仓 |
| 冻结观察 | 买入逻辑被质疑但未确认，或重大信息未完成多源验证。 | 暂停操作，等待信息明朗 |

### 6.6 熔断冷静机制

若连续 3 个交易日触发情绪熔断，冷静期自动延长至 5 个交易日。期间禁止主动交易，只允许执行预先设定且无需临场判断的定投计划。

## 7. 投资大师智慧库需求

### 7.1 智慧映射

| 大师 | 核心智慧 | 系统规则 |
| --- | --- | --- |
| 格雷厄姆 | 安全边际、市场先生 | PE 分位低于 30% 时提示安全边际较高。 |
| 巴菲特 | 能力圈、护城河 | 能力圈外标的拒绝分析；买入前评估护城河。 |
| 利弗莫尔 | 趋势、耐心持有 | 连续新低时禁止盲目抄底；逻辑未破坏时提示保持耐心。 |
| 达利欧 | 风险平价、债务周期 | 检查资产风险贡献；定期提示信贷周期阶段。 |
| 马克斯 | 周期思维、第二层思维 | 市场狂热时展示看空逻辑，市场恐慌时展示看多逻辑。 |
| 彼得·林奇 | 生活常识、分类投资 | 无法清楚描述行业逻辑时建议暂缓；持仓需分类。 |
| 邓普顿 | 极度悲观点 | 情绪极度悲观时提示可研究小仓位试探机会。 |

### 7.2 冲突裁决

| 冲突 | 裁决规则 |
| --- | --- |
| 邓普顿 vs. 利弗莫尔 | 若 PE 分位 <20% 且情绪连续 3 日极度恐惧，可优先考虑小仓位试探。 |
| 格雷厄姆 vs. 利弗莫尔 | 若 PE 分位 <30% 且买入逻辑未破坏，支持分批定投，但降低单次金额。 |
| 巴菲特 vs. 其他观点 | 能力圈外标的，其他正面信号均不得覆盖能力圈规则。 |

### 7.3 P74 内置知识与数据准备度

- 大师智慧、纪律规则、风险 SOP 和标的画像必须以稳定 ID 注册，例如 `master.graham.margin_of_safety`、`discipline.no_single_source_decision`、`risk_sop.evidence_insufficient`、`symbol_profile.510300`。
- 内置知识只能作为纪律原则、规则映射和 LLM 分析上下文，不得被标记或展示为正式市场证据。
- `510300` 本地验收主路径必须映射到 `000300` 跟踪指数画像；未知标的画像必须返回 blocked，不得伪造成 ready。
- ETF/基金/指数使用前必须显式检查数据依赖：标的画像、基金画像、跟踪指数、市场价格、估值分位、流动性、情绪代理、生效规则、正式证据、RAG 索引和 LLM 上下文。
- 任一必需依赖缺失或降级时，系统必须展示安全降级影响，不得声明安全边际、估值高低、交易确认或收益预期已可靠通过。
- LLM 可接收脱敏 readiness 摘要，但不能覆盖最终规则裁决，不能补足正式证据，不能生成自动交易、自动确认或自动规则应用。

## 8. 关键场景 SOP

### 8.1 场景 A：持仓大跌

触发条件：单日或短期跌幅超过 5%。

| 步骤 | 检查项 | 输出 |
| --- | --- | --- |
| 1 | 买入逻辑是否被破坏 | 是：建议卖出并说明破坏点。 |
| 2 | PE 分位位置 | 低估区：可考虑按计划分批补仓；其他区间继续检查。 |
| 3 | 情绪是否极端恐惧 | 是：锁定主动交易，展示历史类似情景数据。 |
| 4 | 综合判断 | 逻辑完好且估值较低：建议按计划分批；估值合理：继续持有。 |

### 8.2 场景 B：持仓大涨

触发条件：短期涨幅超过 15% 或浮盈超过 20%。

| 步骤 | 检查项 | 输出 |
| --- | --- | --- |
| 1 | PE 分位是否 >80% | 是：启动分批止盈流程。 |
| 2 | 浮盈水平 | 20%：建议卖出 30%；30%：建议再卖出 30%。 |
| 3 | 剩余仓位 | 启动 10% 回撤移动止盈。 |
| 4 | 核心-卫星比例 | 卫星资产超标时，止盈资金优先回归核心资产。 |

### 8.3 场景 C：追热点咨询

触发条件：用户主动询问是否可以买入某热点标的。

| 步骤 | 检查项 | 输出 |
| --- | --- | --- |
| 1 | 是否在能力圈内 | 否：拒绝分析并提示纪律约束。 |
| 2 | PE 分位 | >80%：提示追涨风险；<30%：允许进一步分析。 |
| 3 | 仓位上限 | 已达到上限：提示不得突破仓位限制。 |
| 4 | 综合建议 | 即使通过检查，也建议分批建仓并预设退出条件。 |

### 8.4 场景 D：市场恐慌想卖出

触发条件：用户表达强烈恐惧并想清仓。

| 步骤 | 检查项 | 输出 |
| --- | --- | --- |
| 1 | 触发冷静机制 | 暂停主动交易建议。 |
| 2 | 展示客观数据 | 输出 PE 分位、情绪指数、持仓估值。 |
| 3 | 历史回溯 | 展示类似恐慌时刻后的历史表现。 |
| 4 | 理性提醒 | 基于买入逻辑和估值数据给出建议，最终由用户确认。 |

### 8.5 场景 E：宏观灰犀牛事件

| 步骤 | 检查项 | 输出 |
| --- | --- | --- |
| 1 | 是否构成买入逻辑破坏 | 是：建议卖出。 |
| 2 | 是否仅为扰动 | 若日波动率连续 3 日超过历史均值 2 倍，建议减仓而非加仓。 |
| 3 | 情景更新 | 重新运行预期收益评估，下调相关情景概率。 |

### 8.6 场景 F：突发黑天鹅事件

| 步骤 | 检查项 | 输出 |
| --- | --- | --- |
| 1 | 第一时间冻结 | 所有主动交易进入冻结观察 24 小时。 |
| 2 | 信息核查 | 等待至少 2 个 A 级信源的影响评估。 |
| 3 | 重新评估 | 24 小时后按根本规则裁决。 |

## 9. 预期收益评估模块需求

### 9.1 模块目标

为持仓标的提供基于历史规律和当前估值的收益参考区间，并据此触发卖出评估。

### 9.2 设计原则

- 只输出概率估算，不承诺结果。
- 基于历史回测和类似估值水平，而非主观判断。
- 随估值、基本面和市场状态动态更新。
- 当实际涨幅接近预期区间上沿时，自动触发卖出评估。

### 9.3 情景分析

| 情景 | 定义 | 概率依据 |
| --- | --- | --- |
| 乐观情景 | 业绩超预期，估值提升至历史高位 | 历史类似样本比例 |
| 基准情景 | 业绩符合预期，估值维持当前水平 | 样本中出现频率最高路径 |
| 悲观情景 | 业绩低于预期，估值收缩 | 必须展示，用于风险提醒 |

### 9.4 输出格式

每份预期收益评估报告需包含：

- 标的名称与代码。
- 当前日期、当前价格或净值。
- 当前 PE、PB、分位水平。
- 未来 12 个月三类情景收益区间。
- 各情景触发条件和概率；样本不足时按 9.7 降级展示，不输出精确概率。
- 卖出评估建议。
- 止盈、止损、重新评估触发条件。
- 免责声明。

### 9.5 卖出评估触发条件

| 条件 | 动作 |
| --- | --- |
| 当前净值进入乐观情景下沿 | 提示启动移动止盈。 |
| 当前净值突破基准情景上沿 | 提示分批止盈。 |
| 当前净值跌破悲观情景下沿 | 触发买入逻辑核验。 |
| 基准情景中枢下移超过 15% | 提示重新评估买入逻辑并考虑减仓。 |
| 达到用户自定义目标收益 | 提示执行预设止盈计划。 |

### 9.6 压力测试与动态校准

- 定期检查支撑估值的核心假设。
- 若核心假设连续 2 个月低于预期，触发情景下修预警。
- 若实际走势连续 1 个月偏向悲观情景，建议用户手动调整情景概率。

### 9.7 样本门槛

- 历史类似样本数少于 20 个时，不输出精确概率，只输出“样本不足，仅作参考”。
- 历史类似样本数少于 5 个时，不生成收益区间，只输出定性风险提示和需要补充的数据。
- 所有概率输出必须附带样本数量、样本区间和筛选条件。

## 10. 仓位与资产配置需求

### 10.1 核心-卫星策略

| 层级 | 占比 | 标的类型 | 目标 |
| --- | --- | --- | --- |
| 核心资产 | 60%-70% | 红利 ETF、沪深 300 ETF、二级债基 | 稳健增值 |
| 卫星资产 | 20%-30% | AI 主题 ETF、行业 ETF | 争取超额收益，承受较高波动 |
| 现金仓位 | 5%-10% | 货币基金 | 应对风险与机会 |

### 10.2 再平衡规则

| 频率 | 触发条件 | 动作 |
| --- | --- | --- |
| 每季度 | 偏离预设比例 ±15% | 卖出超额部分，买入不足部分。 |
| 动态触发 | 卫星资产超过上限 | 提示调整，止盈资金优先回归核心资产。 |

## 11. 账户状态同步需求

### 11.1 核心机制

采用本地虚拟账户和手动确认机制，账户状态统一存储在 SQLite 中。系统所有建议必须基于用户确认后的账户快照。

### 11.2 持仓录入

用户可手动录入：

- 标的代码
- 标的名称
- 买入日期
- 数量
- 成本价
- 买入逻辑
- 持仓标签

### 11.3 操作确认

当系统建议卖出或调整仓位后，用户需要确认实际执行情况，并录入实际成交数量和价格。系统根据确认结果更新 SQLite 中的账户、持仓和操作记录。

### 11.4 建议数据表

| 表名 | 用途 | 关键字段 |
| --- | --- | --- |
| `portfolio_snapshots` | 保存账户时点快照 | 快照 ID、现金、总资产、仓位比例、快照时间、数据来源 |
| `position_snapshots` | 保存快照下完整持仓集合 | 快照 ID、标的代码、数量、成本价、市值、仓位比例 |
| `positions` | 保存当前持仓聚合状态 | 标的代码、名称、成本价、数量、买入逻辑、持仓状态 |
| `operation_confirmations` | 保存用户对建议的确认动作 | 确认类型、确认状态、确认时间、关联建议编号、备注 |
| `position_transactions` | 保存已手动执行后的交易流水 | 操作类型、标的代码、成交数量、成交价格、成交时间、confirmation_id |
| `decision_records` | 保存系统每次正式建议 | 建议编号、触发规则、证据列表、风险原因码、建议内容、生成时间 |
| `evidence_refs` | 保存建议引用证据 | decision_id、summary_id、source_level、evidence_role、引用原因 |
| `audit_events` | 保存工作流与用户动作审计 | action、node_action、actor、status、before_state、after_state、request_id |

### 11.5 决策依据

每次运行时，系统优先读取 SQLite 中最新且已由用户确认的账户状态。未确认的建议不得改变账户状态。

### 11.6 交易边界

系统默认不接入券商交易 API，不自动下单。所有买入、卖出、再平衡都只生成建议，必须由用户在线下或第三方交易系统中自行执行，并在本系统中手动确认结果。

## 13. 自主学习与进化需求

### 13.1 错误案例库

| 字段 | 说明 |
| --- | --- |
| error_id | 唯一编号 |
| decision_type | 决策类型，如卖出、补仓、加仓、持有 |
| context_snapshot | 决策时市场状态快照 |
| agent_reasoning | Agent 当时的分析逻辑 |
| actual_outcome | 用户标注的实际结果 |
| root_cause_tag | 错误原因标签 |
| lesson_learned | 用户总结的经验 |
| timestamp | 记录时间 |

### 13.2 进化提案类型

- 阈值调整提案：基于多次错误案例建议调整 PE 分位、情绪阈值等参数。
- SOP 增补提案：发现高频未覆盖情景时建议增加检查步骤。
- 大师智慧权重调整提案：根据用户风格建议调整不同流派权重。
- 个人行为模式预警：识别用户高频情绪化决策场景，并提前提示。

### 13.3 守门人审计

任何规则修改在用户确认后，必须经过安全审计官审查。

| 审计项 | 方法 | 处理 |
| --- | --- | --- |
| 是否违反根本规则 | 逐条检查 6 条根本规则 | 违反则否决 |
| 样本是否充分 | 关联案例数少于 3 个 | 警告并建议继续观察 |
| 是否受极端情绪影响 | 检查案例情绪指数 | 标注可能存在情绪偏差 |
| 回测表现是否恶化 | 比较夏普比率、最大回撤等指标 | 明显恶化则否决 |
| 是否与现有规则冲突 | 扫描 SOP 和裁决优先级 | 要求用户解决冲突 |

### 13.4 审计输出

- 通过：规则写入，并记录日志。
- 否决：说明原因，提案作废。
- 需要用户复核：输出修改后版本，等待用户再次确认。

## 14. 数据可追溯与审计日志

系统必须保存从数据输入到建议输出的关键链路，便于复盘和审计。

| 数据 | 保存内容 | 用途 |
| --- | --- | --- |
| 信源证据 | 来源、等级、URL、抓取时间、hash、摘要 | 追溯事实依据 |
| 决策记录 | 触发规则、Agent 观点、裁决依据、置信度 | 复盘建议生成过程 |
| 账户快照 | 每日现金、持仓、市值、仓位比例 | 月度归因和纪律审计 |
| 操作确认记录 | 用户确认动作、成交数量、成交价格、确认时间 | 校准本地账户状态 |
| 审计事件 | action、node_action、用户确认、错误标注、规则提案与应用时间 | 追踪系统执行与进化历史 |

## 15. 与市面 AI 助手的差异

| 维度 | 常见 AI 助手 | 本系统 |
| --- | --- | --- |
| 核心目标 | 提高活跃度或交易频次 | 帮助用户执行纪律 |
| 决策方式 | 模糊多空分析 | 基于阈值和规则的纪律检查 |
| 利益冲突 | 可能存在 | 无交易佣金或流量激励 |
| 卖出信号 | 较少主动提示 | 明确包含止盈、逻辑破坏、估值过高提醒 |
| 情绪管理 | 可能放大交易冲动 | 极端情绪时触发冷静机制 |
| 信源可信度 | 来源不透明 | 信源分级、可追溯 |
| 个性化 | 通用模板 | 基于能力圈、仓位和风险偏好 |
| 可审计性 | 较弱 | 规则、数据、推理可追溯 |
| 自我进化 | 不透明或不可控 | 错误反馈、提案和审计均受控 |

## 17. 验收标准

### 17.1 基础能力验收

- 可录入并读取本地持仓。
- 可记录用户线下操作确认，并更新本地账户快照。
- 可通过 Eino Graph 完成状态快照节点、规则节点和决策记录节点编排。
- 可在 Web 控制台驾驶舱展示账户状态、数据更新时间、纪律状态和触发规则。
- 可根据 PE 分位、现金比例和仓位规则输出纪律提醒。

### 17.2 决策能力验收

- 可获取并存储基础行情和估值数据。
- 可生成每日纪律报告。
- 可通过 Eino Graph 完成价值分析师节点、趋势风控官节点、信息核查节点、规则裁决节点和决策记录节点编排。
- 输出内容包含数据来源、证据链、Agent 观点、触发规则、账户快照版本和规则版本。
- 用户可记录计划、已手动执行、待观察或错误标注。

### 17.3 核心系统验收

- 多 Agent 能生成独立分析观点。
- 裁决机制严格按照优先级执行。
- 单信源重大信息不会触发交易建议。
- 极端情绪状态下不会输出主动交易建议。
- 买入逻辑破坏后，持仓状态能进入只卖不买。

### 17.4 进化机制验收

- 用户可标注错误案例。
- 系统可按周期生成规则优化提案。
- 提案必须经用户确认和守门人审计。
- 审计结果可追溯，所有规则变更有日志。

### 17.5 本地运维与升级验收

- 本地发布/升级检查可报告当前版本、目标版本、备份提醒、迁移文件预检状态、升级后 smoke 命令和安全边界。
- 目标版本缺失时，检查应返回 warning 并继续提供备份与 smoke checklist；不修改本地数据。
- 发布/升级诊断 JSON 与脚本摘要只包含脱敏状态、命令模板、计数、占位路径和安全说明，不包含完整 API key、私有绝对路径、raw SQL、完整 prompt、raw HTTP、私钥、供应商原始响应、账户密钥或券商凭证。
- 本地安装诊断脚本只有在用户显式启用 release/upgrade 检查时才纳入该步骤；默认诊断行为保持不变。
- 发布/升级检查不得自动下载、升级、迁移、备份、恢复、修复、覆盖真实库、标记安装健康、执行交易、应用规则、代为确认、调用 LLM、访问公网端点或承诺收益。

## 18. 风险与合规说明

- 本系统仅作为个人投资决策辅助工具。
- 所有输出均不构成投资建议或收益保证。
- 最终买卖决策由用户本人承担。
- 系统必须保留完整免责声明。
- 涉及账户、交易、数据源 API 的信息应仅保存在本地或用户授权环境中。
- 系统默认不接入券商交易 API，不自动执行任何交易。
