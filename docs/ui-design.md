# Investment Agent 前端 UI 设计规范

> 文档版本：v1.0  
> 最后更新：2026-05-21  
> 适用范围：Investment Agent 的 Web 控制台、每日纪律报告、决策咨询、持仓管理、错误标注、规则提案审批与系统审计界面。  
> 设计定位：一个冷静、克制、可审计的个人投资纪律工作台，不做刺激交易的行情软件。  
> 数据契约：页面字段与 API DTO 的映射以 `docs/frontend-contract.md` 为准，HTTP API 以 `docs/api.md` 为准。

## 1. 设计目标

### 1.1 产品目标

前端界面需要帮助用户完成三件事：

- 看清当前账户、仓位、估值、风险状态。
- 理解决策建议背后的规则、证据和裁决过程。
- 在关键时刻减少情绪化操作，辅助用户执行既定纪律。

### 1.2 设计原则

| 原则 | UI 要求 |
| --- | --- |
| 冷静优先 | 不使用刺激性涨跌动效，不制造紧张感和交易冲动。 |
| 纪律可见 | 所有建议都展示触发规则、证据来源和状态判断。 |
| 决策可追溯 | 每份报告都能查看信源、Agent 观点、裁决依据和日志。 |
| 风险前置 | 风险、约束、禁止项的视觉层级高于收益想象。 |
| 用户确认 | 所有操作建议都进入确认流程，不暗示系统可代替用户执行交易。 |
| 信息分层 | 日常监控简洁，深度证据可展开查看。 |

## 2. 视觉方向

### 2.1 设计气质

采用 **冷静审计工作台** 风格：

- 克制、专业、清晰。
- 视觉上接近风控系统、研究终端、审计台账，而不是券商交易软件。
- 不以红绿涨跌刺激用户，而以状态、规则、证据和行动边界组织信息。

### 2.2 关键词

- Calm
- Disciplined
- Auditable
- Evidence-first
- Rule-based
- Personal cockpit

### 2.3 避免的视觉方向

- 高频交易终端风格：密集闪烁、强刺激红绿、实时跳动价格墙。
- 社交媒体风格：情绪化标题、榜单焦虑、热点诱导。
- 游戏化理财风格：成就、连胜、收益炫耀。
- 通用 SaaS 模板风格：缺少投资纪律和审计感。

## 3. 核心界面模型：Agent 决策驾驶舱

### 3.1 主窗口定位

系统主窗口不是聊天窗口，也不是普通后台首页，而是 **Agent 决策驾驶舱**。

主窗口承担整个产品的核心体验：

- 展示当前账户、持仓、市场和纪律状态。
- 展示是否触发根本规则、SOP 或风险红线。
- 展示 Agent 的结构化裁决报告。
- 展示证据摘要、规则命中和账户快照。
- 提供用户确认、待观察、错误标注等入口。

咨询输入框只是触发 Agent 工作流的入口，不能成为界面中心。界面中心必须是结构化裁决报告。

### 3.2 主窗口三栏结构

桌面端主窗口采用三栏结构：

| 区域 | 内容 | 作用 |
| --- | --- | --- |
| 左侧：导航与系统状态 | 一级导航、账户状态、数据更新时间、当前模式 | 让用户知道系统在哪里、数据是否新鲜、是否处于冻结或观察状态。 |
| 中间：Agent 裁决工作区 | 当前状态、触发规则、今日建议、多 Agent 观点、最终裁决、用户确认区 | 承载核心判断和用户动作。 |
| 右侧：证据与规则面板 | 信源摘要、规则命中、账户快照、审计状态、样本数量 | 支撑裁决的证据、规则和上下文。 |

### 3.3 主窗口默认模式

主窗口默认打开 **今日纪律模式**，即用户每天最先看到的纪律巡检结果。

今日纪律模式的第一屏固定展示：

1. 当前纪律状态。
2. 触发红线与禁止事项。
3. 今日建议。
4. 账户与持仓摘要。
5. 证据摘要。证据数据完整时展示信源数量、最高信源等级和多源验证状态；证据不足时展示信息不足提示。

### 3.4 咨询模式

用户主动提问时，主窗口进入 **决策咨询模式**。

展示方式必须从“输入问题”过渡到“裁决报告”，固定展示：

1. 用户问题。
2. 能力圈检查。
3. 信息核查。
4. 价值分析师观点。
5. 趋势与风控官观点。
6. 规则裁决。
7. 最终建议与用户确认。

### 3.5 驾驶舱页面状态

驾驶舱必须显式展示当前页面状态，状态来源以 `docs/frontend-contract.md` 的页面状态契约为准。

| 状态 | 触发条件 | 展示重点 | 允许动作 |
| --- | --- | --- | --- |
| 首次使用 | 未录入账户或持仓 | 引导录入账户、持仓、能力圈 | 录入持仓、配置设置 |
| 正常 | 数据完整且未触发高危规则 | 今日纪律状态、账户摘要、证据摘要 | 查看详情、查看证据 |
| 信息不足 | 行情过期、证据不足、索引不可用 | 缺失数据、更新时间、暂停原因 | 刷新数据、查看缺失项 |
| 冻结观察 | 多源验证不足、突发事件待核查 | 冻结原因、证据状态、等待条件 | 查看证据、标记待观察 |
| 高危 | 逻辑破坏、规则缺失、记录失败 | 禁止事项、触发规则、风险说明 | 查看规则、记录线下结果 |

### 3.6 其他页面定位

其他页面不是主体验，而是支撑 Agent 裁决的资料库、配置台和审计台。

| 页面类型 | 页面 | 定位 |
| --- | --- | --- |
| 数据管理页 | 持仓、设置 | 管理账户状态、能力圈、仓位上限、通知等基础数据。 |
| 证据页 | 情报与证据 | 查看信源、摘要、VecLite 命中块和多源验证状态。 |
| 规则页 | 规则与纪律 | 查看根本规则、SOP、阈值和大师智慧映射。 |
| 审计页 | 复盘与审计、错误案例、规则提案、规则版本 | 追踪建议、用户确认、错误标注、规则进化和审计事件。 |

## 4. 信息架构

### 4.1 一级导航

| 导航 | 目标 |
| --- | --- |
| 今日纪律 | 展示每日账户状态、风险红线、纪律建议。 |
| 持仓 | 管理账户、持仓、成本、买入逻辑和状态机。 |
| 决策咨询 | 用户主动提问后生成结构化分析。 |
| 情报与证据 | 查看信源、情报摘要、交叉验证结果。 |
| 规则与纪律 | 查看根本规则、SOP、阈值和大师智慧映射。 |
| 复盘与审计 | 查看历史建议、操作确认、错误案例、规则提案、规则版本和审计事件。 |
| 设置 | 管理能力圈、仓位上限、现金比例、数据源和通知。 |

### 4.2 关键页面

#### 今日纪律页

用途：用户每天打开系统后的主页面。

首屏固定顺序展示：

1. 纪律状态：正常、观察、高危、冻结、只卖不买。
2. 纪律红线：仓位上限、现金底线、逻辑破坏、流动性风险。
3. 今日建议：持有、暂停定投、分批止盈、冻结观察等。
4. 账户概览：总资产、现金比例、核心资产、卫星资产、仓位状态。
5. 证据摘要：展示行情、估值、主要信源和多源验证状态；证据不足时展示缺失项和暂停原因。

核心模块：

- 账户概览：总资产、现金比例、核心资产、卫星资产、仓位状态。
- 市场状态：PE 分位、PB 分位、情绪状态、资金状态。
- 纪律红线：仓位上限、现金底线、逻辑破坏、流动性风险。
- 今日建议：持有、暂停定投、分批止盈、冻结观察等。
- 证据摘要：展示本次判断引用的主要数据、信源等级、发布时间和多源验证状态；证据不足时展示缺失项。

#### 持仓页

用途：管理用户确认后的账户状态。

核心模块：

- 持仓列表。
- 持仓详情。
- 买入逻辑记录。
- 持仓状态机：正常持有、只卖不买、冻结观察。
- 成本、数量、市值、浮盈、仓位比例。
- 操作确认记录。

#### 决策咨询页

用途：用户主动询问某个标的或场景。

核心模块：

- 咨询输入框。
- 能力圈检查。
- 信息核查结果。
- 价值分析师观点。
- 趋势与风控官观点。
- 规则裁决结果。
- 灰度建议和禁止事项。
- 证据链展开区。

#### 决策详情页

用途：展示一条建议从输入、证据、分析到裁决的完整链路。

核心模块：

- 建议摘要：建议编号、生成时间、当前状态、用户确认状态。
- 账户快照：建议生成时的现金、持仓、仓位比例、持仓状态。
- 触发规则：命中的根本规则、SOP 步骤、市场状态阈值。
- Agent 观点：价值分析师、趋势与风控官的独立观点。
- 裁决链：裁决优先级命中情况和最终建议。
- 证据链：信源等级、发布时间、摘要、原文入口、时效权重。
- 用户确认区：记录已执行、标记待观察、标记错误。

#### 情报与证据页

用途：查看 RAG 与信源分级结果。

核心模块：

- 情报列表。
- 信源等级筛选。
- 发布时间与时效权重。
- 多源验证状态。
- 原文链接、摘要、hash。
- VecLite 检索命中块。

#### 规则与纪律页

用途：展示系统当前执行的规则。

核心模块：

- 根本规则。
- 裁决优先级。
- 市场状态阈值。
- 移动止盈规则。
- 关键场景 SOP。
- 大师智慧映射。

#### 复盘与审计页

用途：复盘系统建议和用户行为。

核心模块：

- 决策记录。
- 操作日志。
- 账户快照。
- 错误案例库。
- 进化提案。
- 守门人审计结果。
- 正式规则版本。

## 5. Agent 展示流程设计

### 5.1 标准展示主线

每次 Agent 输出建议时，界面必须展示以下主线：

1. 当前状态：账户、持仓、市场、纪律状态。
2. 触发规则：根本规则、SOP、阈值、禁止事项。
3. 规则裁决：裁决优先级命中情况和最终建议。
4. 用户确认：记录已执行、待观察、标记错误、查看证据。
5. 证据链：展示信源等级、发布时间、时效权重和多源验证状态；证据不足时展示缺失原因。
6. 多 Agent 观点：展示价值分析师、趋势与风控官、信息核查结果。

### 5.2 展示优先级

| 优先级 | 展示内容 | 原则 |
| --- | --- | --- |
| 1 | 纪律状态和禁止事项 | 先告诉用户能不能行动。 |
| 2 | 触发规则和裁决结果 | 先展示规则判断，再展示分析观点。 |
| 3 | 证据摘要 | 展示可信来源摘要，细节可展开；证据不足时明确提示缺失项。 |
| 4 | 多 Agent 观点 | 作为分析材料，不作为最终裁决。 |
| 5 | 历史记录 | 放在详情或审计页。 |

### 5.3 禁止的展示方式

- 不把大模型回答作为页面主角。
- 不用聊天气泡承载关键建议。
- 不隐藏规则裁决。
- 不把证据链放到多层跳转之后。
- 不用收益预期覆盖风险和纪律状态。

## 6. 用户流程

### 6.1 每日查看流程

1. 用户进入今日纪律页。
2. 系统展示账户、市场状态和纪律红线。
3. 用户查看今日建议。
4. 如有风险或建议，用户可展开证据信息；证据不足时查看缺失项和暂停原因。
5. 若建议涉及操作，用户只能记录“计划”或“已手动执行”，系统不提供自动交易入口。

### 6.2 决策咨询流程

1. 用户输入问题，例如“这个标的还能不能持有”。
2. 系统检查能力圈和持仓状态。
3. 信息核查官检索 S/A/B 级信源。
4. 价值分析师与趋势风控官分别给出观点。
5. 规则裁决模块输出建议。
6. 用户查看证据和规则后，自行决定是否在线下操作。
7. 用户可记录实际执行结果。

### 6.3 错误标注流程

1. 用户进入历史决策记录。
2. 选择一条建议，标记结果：正确、错误、待观察。
3. 输入实际结果、原因、个人教训。
4. 系统写入错误案例库。
5. 后续由进化提案引擎定期归纳。

### 6.4 规则提案审批流程

1. 系统生成规则优化提案。
2. 用户查看提案来源、关联案例、回测结果。
3. 用户确认后，守门人审计。
4. 审计结果为通过、否决、需要用户复核。
5. 审计通过后进入 `pending_final_confirm`，由用户最终确认。
6. 用户最终确认后写入 `rule_versions`，提案状态变为 `applied`，并生成 `audit_events`。

## 7. 页面布局规范

### 7.1 桌面端布局

固定布局：左侧导航 + 顶部状态条 + 主内容区。

- 左侧导航宽度：240px 左右。
- 顶部状态条展示：当前日期、数据更新时间、账户状态、系统模式。
- 主内容区采用 12 栅格布局。
- 今日纪律页第一屏展示核心状态，不让用户滚动后才看到风险提示。

### 7.2 移动端布局

移动端优先支持查看和确认，不承载复杂复盘。

底部导航固定包含：

- 今日
- 持仓
- 咨询
- 审计
- 设置

移动端页面应减少并列表格，改为卡片、折叠区和时间线。

### 7.3 信息层级

页面层级固定为：

1. 当前状态。
2. 是否触发规则。
3. 建议动作或禁止动作。
4. 证据摘要。
5. 详细数据和历史记录。

## 8. 核心组件规范

### 8.1 DisciplineStatusCard

用途：展示今日纪律状态。

字段：

- 状态名称：正常、观察、冻结、只卖不买、高危。
- 触发规则。
- 风险等级。
- 建议动作。
- 更新时间。

### 8.2 RuleTriggerBadge

用途：标识触发的规则。

示例：

- R-1 多源验证不足
- R-3 仓位上限
- R-4 买入逻辑破坏
- R-6 流动性风险

### 8.3 EvidenceCard

用途：展示证据来源。

字段：

- 信源等级。
- 来源名称。
- 发布时间。
- 时效权重。
- 摘要。
- 原文链接。
- hash。

### 8.4 AgentOpinionPanel

用途：展示分析师观点。

每个观点必须包含：

- 结论倾向。
- 关键依据。
- 风险提示。
- 置信度。
- 引用证据。

### 8.5 DecisionVerdictPanel

用途：展示规则裁决结果。

字段：

- 最终建议。
- 裁决优先级命中情况。
- 禁止项。
- 可选行动。
- 需要用户确认的事项。

### 8.6 PositionStateTag

用途：展示持仓状态机。

状态：

- 正常持有：`normal`
- 只卖不买：`sell_only`
- 冻结观察：`frozen_watch`

状态标签必须附带说明，不只使用颜色。

### 8.7 AuditTimeline

用途：展示审计链路。

节点：

- 建议生成
- 用户确认
- 错误标注
- 提案生成
- 守门人审计
- 规则更新

### 8.8 ScenarioReport

用途：展示预期收益评估。

模块：

- 当前估值。
- 样本数量。
- 样本区间。
- 乐观、基准、悲观情景。
- 样本不足降级提示。
- 卖出评估触发点。

### 8.9 PortfolioTable

用途：展示持仓明细。

字段：

- 标的名称与代码。
- 持仓状态。
- 数量。
- 成本价。
- 当前价格。
- 市值。
- 浮盈。
- 仓位比例。
- 买入逻辑摘要。

### 8.10 DecisionRecordTable

用途：展示历史建议记录。

字段：

- 建议编号。
- 生成时间。
- 标的或场景。
- 最终建议。
- 触发规则。
- 用户确认状态。
- 后续标注结果。

### 8.11 RuleChangeDiff

用途：展示规则提案或规则变更前后的差异。

字段：

- 规则名称。
- 变更前内容。
- 变更后内容。
- 变更原因。
- 关联错误案例。
- 守门人审计结论。

### 8.12 SourceVerificationPanel

用途：展示多源验证状态。

字段：

- 验证结论：已满足、未满足、仅作背景。
- 独立信源数量。
- 最高信源等级。
- 最新发布时间。
- 时效权重。
- 证据摘要列表。

### 8.13 ConfirmActionPanel

用途：展示用户确认区。

可用动作：

- 记录已执行。
- 标记为待观察。
- 提交错误标注。
- 确认提案进入审计。

禁止出现自动交易、代为执行、一键买卖类动作。

## 9. 状态与颜色规范

### 9.1 状态颜色

颜色只服务于状态识别，不用于刺激用户交易。

| 状态 | 颜色倾向 | 用法 |
| --- | --- | --- |
| 正常 | 冷灰、低饱和蓝 | 日常安全状态。 |
| 观察 | 琥珀色 | 需要关注但不紧急。 |
| 高危 | 深红棕 | 纪律红线或高风险状态。 |
| 冻结 | 深蓝灰 | 暂停操作、等待验证。 |
| 只卖不买 | 紫灰或酒红 | 买入逻辑破坏后的限制状态。 |
| 信息不足 | 中性灰 | 数据或样本不足。 |

### 9.2 涨跌颜色

- 不使用大面积高饱和红绿。
- 涨跌数字可使用低饱和颜色，并配合正负号。
- 风险提示优先于收益数字。
- 收益展示不得使用庆祝动效。

### 9.3 禁止只靠颜色表达状态

所有状态必须同时有文字、图标或标签说明，满足可访问性要求。

## 10. 字体与排版

### 10.1 字体方向

使用具备审计感和阅读稳定性的字体组合：

- 标题：较窄、冷静、有技术感的字体。
- 正文：高可读性中文字体。
- 数字：等宽或半等宽字体，用于金额、比例、阈值、日期。

### 10.2 字号层级

| 层级 | 用途 |
| --- | --- |
| H1 | 页面标题。 |
| H2 | 页面核心区块。 |
| H3 | 卡片标题、子模块。 |
| Body | 正文说明。 |
| Caption | 数据来源、更新时间、审计说明。 |
| Mono | 金额、比例、规则编号、hash。 |

### 10.3 排版要求

- 金额、百分比、规则编号使用等宽数字。
- 长段分析必须分段展示。
- 决策建议不使用过度强调的大字标题。
- 数据来源和更新时间始终可见。

## 11. 文案规范

### 11.1 语气

- 冷静、克制、明确。
- 少用煽动性表达。
- 不暗示确定性收益。
- 不制造错过机会的焦虑。

### 11.2 文案格式

输出结构：

```text
当前状态：观察区
触发规则：PE 分位 63%，处于观察区
建议：暂停新增买入，继续监控
依据：估值未进入低估区，且未发现买入逻辑破坏
注意：本建议不构成投资建议，最终决定由用户本人作出
```

### 11.3 禁止文案

- “马上买入”
- “必涨”
- “错过就没了”
- “稳赚”
- “满仓机会”
- “系统已替你判断”

## 12. 可追溯设计规范

### 12.1 每条建议必须展示

- 建议编号。
- 生成时间。
- 触发规则。
- 账户快照版本。
- 使用的数据源。
- Agent 观点摘要。
- 规则裁决结果。

### 12.2 证据展开层级

采用三级展开：

1. 证据摘要。
2. 信源列表。
3. 原文与 hash。

### 12.3 审计日志展示

审计日志采用时间线展示，每个节点必须有：

- 时间。
- 操作者：系统、用户、守门人。
- 动作。
- 结果。
- 关联记录 ID。

## 13. 表单与确认规范

### 13.1 手动持仓录入

字段要求：

- 标的代码。
- 标的名称。
- 买入日期。
- 数量。
- 成本价。
- 买入逻辑。
- 资产标签。

买入逻辑不能为空。

### 13.2 操作确认

操作确认区区分“记录计划”和“已手动执行”：

| 动作 | 含义 | 是否更新账户 | 必须提示 |
| --- | --- | --- | --- |
| 记录计划 | 用户只是记录未来可能执行的计划 | 否 | 该操作不会改变本地账户 |
| 已手动执行 | 用户已经在线下或第三方交易系统完成交易 | 是 | 系统不自动交易，只记录用户已完成的事实 |
| 标记待观察 | 用户暂不确认结果，后续继续追踪 | 否 | 该建议仍处于待观察状态 |
| 标记错误 | 用户认为建议结果有误或需要复盘 | 否 | 将写入错误案例库 |

当用户确认已在线下执行交易时，需要录入：

- 操作类型。
- 标的代码。
- 成交数量。
- 成交价格。
- 成交时间。
- 关联建议编号。

确认页面必须展示：系统不自动交易，本记录仅用于更新本地虚拟账户。

### 13.3 规则提案确认

确认前必须展示：

- 提案来源。
- 关联错误案例。
- 回测结果。
- 是否违反根本规则。
- 守门人审计结论。
- 变更前后对比。

## 14. 空状态与异常状态

### 14.1 空状态

空状态不应过度营销，只说明下一步。

示例：

```text
暂无持仓。请先录入账户现金和持仓，系统才能生成纪律报告。
```

### 14.2 数据不足

示例：

```text
历史类似样本不足 20 个，本次不输出精确概率，仅展示定性风险提示。
```

### 14.3 信源不足

示例：

```text
当前只有 1 个 A 级信源，未满足多源验证规则。系统暂停生成交易类建议。
```

## 15. 可访问性与响应式

- 所有状态不能只依赖颜色。
- 关键按钮必须有明确文本。
- 证据、规则、建议需支持键盘访问。
- 图表需配套文字说明。
- 移动端优先保证查看、确认、标注流程可用。

### 15.1 P62 设计系统与可访问性加固

- 前端基础控件应优先复用 Button、Field、StatusBadge、PageHeader、SummaryCard、DetailSection、ResponsiveTable、EmptyState、ErrorState。
- 状态 tone 统一为 success、warning、danger、degraded、unknown、readonly、blocked；状态必须有可读文本，不得只依赖颜色。
- 主导航、移动菜单、表单控件、折叠详情区和关键本地按钮必须具备键盘路径。
- 折叠区按钮必须维护 `aria-expanded`；表单控件必须关联 label、hint 或 error；表格必须有 caption 或可访问名称。
- 390px、768px、1280px 三档 viewport 均不得出现页面级横向滚动；二维表格、JSON、日志或诊断文本只能在明确局部容器滚动。

## 17. 前端实现规范

### 17.1 技术规范

Web 前端固定采用：

- React + Vite + TypeScript。
- 图表采用 ECharts。
- 表格支持筛选、排序、展开证据。
- 状态管理按页面和业务模块组织，避免全局状态滥用。
- 所有金额、比例、日期格式统一封装。

### 17.2 界面数据展示约定

前端必须把系统内部数据转成人能理解的文字，不直接暴露技术字段。字段映射、状态映射和组件数据来源以 `docs/frontend-contract.md` 为准。

必须展示的信息：

- 数据更新时间：例如“数据更新于 2026-05-21 16:30”。
- 账户快照时间：例如“基于今日 16:00 的账户快照”。
- 当前规则版本：例如“当前规则版本 v3.0”。
- 建议生成时间：例如“本建议生成于 2026-05-21 16:35”。
- 证据来源数量：例如“引用 3 个独立信源，其中 2 个为 A 级以上”。
- 审计状态：例如“未审计、审计通过、审计否决、需要用户复核、等待最终确认”。
- 用户确认状态：例如“未确认、已记录执行、待观察、已标记错误”。

展示要求：

- 建议编号必须转换成易读形式，例如“2026-05-21 第 1 条建议”。
- hash、内部 ID 默认折叠，只在证据详情或审计详情中展示。
- 技术字段不得成为页面主信息。

### 17.3 安全提醒

前端不得出现自动交易入口。所有操作按钮应使用以下语义：

- 记录已执行
- 标记为待观察
- 查看证据
- 提交错误标注
- 确认提案进入审计

避免使用：

- 立即买入
- 一键卖出
- 自动跟随
- 代为执行

## 18. 设计验收标准

- 用户能在 30 秒内理解今日是否触发纪律红线。
- 每条建议都能找到触发规则和证据来源。
- 极端行情下界面不刺激用户进行冲动操作。
- 持仓状态机清晰可见。
- 规则提案审批过程完整可追溯。
- 移动端能完成查看、确认、标注三类核心动作。
- 无自动交易入口。
